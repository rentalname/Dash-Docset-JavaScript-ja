    一旦ビルドシステムのセットアップが完了して、コードの最初のプルと構成を実行したら、Boot to Gecko をビルドすることができます。

    コードを更新する

    もし B2G をビルドするのが全く初めてという場合でなければ、ビルドを始める前に最新版のコードをプルしたいかもしれません。これを行うには B2G ツール群と、それに依存関係があるものを、以下の二つのコマンドを使用して、両方とも更新する必要があります :

    git pull
./repo sync -d

    "-d"オプションは指定したプロジェクトをマニフェストのリビジョン(つまりデフォルトのメインストリームのリポジトリのバージョン)に戻すかを切り替えます。これは、リポジトリ(と関連ソース)に変更を加えているが、マスターのバージョンに一時的に戻したい時に有用です。途中段階や作業中のディレクトリの変更はそのままにされます (下記のSOの質問 を見れば全ての説明があります)。ソースコードに触れていない場合、下記のように通常のコマンドも使用できます:

    git pull
./repo sync

    名前を指定することで、作成するターゲットのリポジトリを指定して更新することができます :

    ./repo sync gaia

    repo コマンドには、興味深い、その他のオプションも利用可能です。repo help でたくさんの情報を得ることができます。

    ビルド

    注意 : ビルドの前に、ビルドのカスタマイズをする .userconfig ファイルをセットアップしたいかもしれません。詳細は .userconfig ファイルのカスタマイズを参照してください。

    Boot to Gecko をビルドするには、単純に build.sh ツールを使用してください :

    cd B2G
./build.sh

    もう一回コーヒーブレークの時間です。昼寝もできるかもしれません（特にこれが初回のビルドなら、Core i7の8GB RAMで30分くらいかかります。）。前のページの設定ステップの中で、もしAndroidシステムが事前に展開されたディレクトリを使っている場合、 build.shを実行する前にANDROIDFS_DIRを設定すべきです。

    Note: 新しいビルドを電話機に焼くための詳しい情報は、Boot to Gecko をモバイルデバイスにインストールするを見て下さい。

    特定モジュールのビルド

    例えば Gecko などの、特定のモジュールだけビルドしたい場合は、名前で指定することができます :

    ./build.sh gecko

    一つのアプリケーションだけを更新するには、BUILD_APP_NAME 環境変数を使用することで gaia モジュールのみをビルドすることができます :

    BUILD_APP_NAME=calendar ./build.sh gaia

    ビルド可能なモジュールのリストを取得するには、こうやってください :

    ./build.sh modules

    使用するコアプロセッサの数を設定する

    デフォルトでは、B2G ビルドスクリプトは、システムのコア数に 2 を足したものを、実行する並列タスクの数として使用します。これを build.sh を実行するときに -j パラメータを指定することで変更できます。 これはバックグラウンドでビルド中に、他のことにシステムを使っていて、CPU 負荷を少し減らしたい場合に有効でしょう。また、ビルドに問題がある場合も、実行しているのが一度に 1 つのタスクだけなら、ビルドプロセスからのエラー出力を読むのが楽ですよね！

    例えば、並行処理を 2 つ使用してビルドするには :

    ./build.sh -j2

    しかし、この設定の最も一般的な使い方は、ビルドが並列処理を全く行わないようにすることです。処理のアウトプットが大変読みやすくなり、ビルドの問題が整理しやすくなります。これを実行するには :

    ./build.sh -j1

    複数ロケールのビルド

    複数ロケールのビルドを作成するには、下記に従います:

    Gaia

    1. どのGaia言語ファイルを使うかを決定します。我々はlocales/languages_dev.json とlocales/languages_all.jsonをGaia言語ファイルとして使っています。
    2. http://hg.mozilla.org/gaia-l10n からディレクトリへ適切なロケールを複製します。我々はgaia-l10n/を使います。locales/ディレクトリを使う事もできます。言語ファイルに記載されているロケールごとのrepoをクローンする必要があります。
    3. あなたの環境で、ステップ2のディレクトリの絶対パスをLOCALE_BASEDIRに設定します。ステップ1の絶対パスをLOCALES_FILEに設定します。
    4. その他、デフォルトロケールを設定したい場合、GAIA_DEFAULT_LOCALEに設定します。
    cd gaia/locales/
hg clone https://hg.mozilla.org/releases/gaia-l10n/v1_2/es
    環境変数の設定:
     
    export LOCALE_BASEDIR=$PWD/locales
export LOCALES_FILE=$PWD/locales/languages_dev.json
export GAIA_DEFAULT_LOCALE=es

    languages-dev.jsonをあなた自身のlanguages-own.jsonのパスに置換できます。このように。ロケールディレクトリに複製されたrepo毎にエントリーを追加する必要があるでしょう:

    {
  "en-US"     : "English (US)",
  "es"        : "Español"
}

    バグ 884752 が落ち着いたため(2013年11月)、キーボードレイアウトを追加するため(Gaia内で入手可能なら)GAIA_KEYBOARD_LAYOUTSパラメータを使用できます。

    Bug: https://bugzilla.mozilla.org/show_bug.cgi?id=884752

    Layouts: https://github.com/mozilla-b2g/gaia/tree/v1.2/keyboard/layouts

    例えば、スペイン語とイタリア語のキーボードレイアウトを追加するには先のコマンドに下記を付け足します。

      GAIA_KEYBOARD_LAYOUTS=en,es,it

    この時点で初めてGaiaを電話機に焼く準備ができます、電話機を接続し、リモートデバッグが有効かSettings > Device Information > More Information > Developerにてチェックします。これをたった一度だけでできます、Gaiaビルド用に、REMOTE_DEBUGGER=1という設定をオンにできるおかげで:

     make clean && make production LOCALES_FILE=locales/languages-own.json

    特定ブランチのサポートするロケール全てを取得や更新したい場合、このスクリプトを使います。

    Gecko

    1. どのGecko言語ファイルを使うかを決定します。我々はb2g/locales/all-localesをGecko言語ファイルとして使っています。
    2. 適切なロケールをディレクトリ(ひょっとして gecko-l10n/)にクローンします。.
      • For mozilla-central, use http://hg.mozilla.org/l10n-central/
      • For mozilla-aurora, use http://hg.mozilla.org/releases/l10n/mozilla-aurora/
      • For mozilla-beta or mozilla-b2g18 or mozilla-b2g26_v1_2, use http://hg.mozilla.org/releases/l10n/mozilla-beta/
    3. compare-localesをクローンします。gitを希望する場合、RELEASE_0_9_7タグを使うようにします。

    4. あなたの環境で、ステップ2で指定したパスの絶対パスを L10NBASEDIR にセットします。ステップ1のロケールをスペースで区切って MOZ_CHROME_MULTILOCALE にセットします。

      compare-locales/scripts ディレクトリを PATH に追加し、compare-locales/lib を PYTHONPATH に追加します。

    5. 例えば、 
      export L10NBASEDIR=$PWD/gecko-l10n
export MOZ_CHROME_MULTILOCALE="ja zh-TW"
export PATH="$PATH:$PWD/compare-locales/scripts"
export PYTHONPATH="$PWD/compare-locales/lib"

      いったん上記の設定が終わると、build.shを実行できます。

      同じく .userconfigを使用します:

      ## Gaia
export GAIA_DEFAULT_LOCALE=es
export LOCALE_BASEDIR=/b2g/gaia-l10n-transifex-v1.2/
export LOCALES_FILE=/b2g/languages_propio.json

## Gecko
export L10NBASEDIR='/b2g/gecko-l10n-v1.2'
export MOZ_CHROME_MULTILOCALE="es-ES eu ca gl pt-BR"
export PATH="$PATH:/b2g/compare-locales/scripts"
export PYTHONPATH="/b2g/compare-locales/lib"

    Gaiaの言語とGeckoとで対応があります。例えば、Gaiaではスペイン語は"es"ですがGeckoは"Spanish of Spain" (es-ES)になっています。

    これらの方法は、洗練させていくうちに変わるかもしれません。

    既知のエラー

    "Build failed!"

    汎用的な"Build failed"というメッセージが出た時、いつも電話機をコンピューターに接続し直すべきです。色々な理由で電話機がアンマウントされます。

    Note that configuring and building B2G for Keon DOESN'T WORK on Mac. You'll need to use Linux to build for this device.

    Mountain Lion固有のビルドエラー

    1. OS X 10.8 "Mountain Lion" (Xcode 4.4.1 以降)をビルドに使用して、下記のエラーが出た場合:

    external/qemu/android/skin/trackball.c:130:25: error: 'M_PI' undeclared (first use in this function)
    B2G/external/qemu/Makefile.android のファイルを編集し、78行にこう挿入します:
    MY_CFLAGS += -DM_PI=3.14159265358979323846264338327950288   #/* B2G_fix: not finding M_PI constant */
    2. "Mountain Lion"上で./build.sh 実行中に、下記のエラーが出た場合:
    /System/Library/Frameworks/IOKit.framework/Headers/usb/USB.h:797:9: error: too many #pragma options align=reset

    /System/Library/Frameworks/IOKit.framework/Headers/usb/USB.h の中で '#pragma options align=reset' の箇所を '#pragma pack()' に置換して下さい。

    Undefined symbols "_sqlite3_androidopt_handle_pragma" and "_sqlite3_androidopt_open"

    このエラーは、OS X 10.7 以降でXcode 4.5 以降にてビルドしている時に出ます。これを直すには、external/sqlite/dist/Android.mk のファイルに https://groups.google.com/forum/#!msg/android-building/yAfPyUqCsiQ/7zvICk4GWjYJ のパッチをあてます。

    KeyedVector.h:193:31: error: indexOfKey was not declared in this scope

    このエラーは gcc のバージョンが新しすぎる場合に発生します。gcc/g++/g++-multilib 4.6.x バージョンをインストールしてください。

    詳細な情報については、.userconfigファイルをカスタマイズするを参照してください。

    Community Note: It is possible to use gcc 4.7.x with slight modifications to the B2G code (gcc will guide you) but you won't get any help! Neither with modifying the code nor with bugs you encounter.

    arm-linux-androideabi-g++: Internal error: Killed (program cc1plus)

    このエラーが表示されたら、恐らく空きメモリが不足しています。./build.sh を実行する前に十分な空きメモリを確保してください。うまく実行させるには 4GB の RAM が必要です。

    "...is referenced by DSO" error

    エミュレーターのビルド時に、このようなエラーが出た場合:
     /usr/bin/ld: out/host/linux-x86/obj/EXECUTABLES/triangleCM_intermediates/triangleCM: hidden symbol `_XGetRequest' in out/host/linux-x86/obj/STATIC_LIBRARIES/libSDL_intermediates/libSDL.a(SDL_x11dyn.o) is referenced by DSO.

    binutilskのいくらかのバージョンでこれが出ます。Debian安定版では、binutils-gold パッケージをインストールして gold リンカを使えます。注意点としてgold リンカはbinutils によってインストール済ですがデフォルトでは使用されず、binutils-goldは単にそうします。

    ビルドシステムのテスト中にビルドエラーが発生した場合

    ビルドシステムのビルド後のテスト中に、時々（特にビルドツールやオペレーティングシステムの更新後に）このような奇妙なエラーが発生することがあります :

    Generating permissions.sqlite...
test -d profile || mkdir -p profile
run-js-command  permissions
WARNING: permission unknown:offline-app
WARNING: permission unknown:indexedDB-unlimited
build/permissions.js:122: NS_ERROR_UNEXPECTED: Component returned failure code: 0x8000ffff (NS_ERROR_UNEXPECTED) [nsIPermissionManager.add]
make[1]: *** [permissions] Error 3
make: *** [gaia/profile.tar.gz] Error 2

    このような状況になったら、gaia/xulrunner-sdk ディレクトリを削除して、コードを再度プルしてみてください :

    rm -r gaia/xulrunner-sdk

    これで、ビルドシステムが自動的に取得した、ダウンロード、プリコンパイル済みの XULRunner のコピーが削除され、次回のビルド時に、XULRunner の新しいコピーが自動的に取得されます。

    Cannot fetch platform/libcore

    Nexus SでB2Gをビルドするconfigureの時(./config.sh nexus-s)にlibcore関連のエラーが出る場合、これを提供するlinaro gitの問題です。これを修正するには、下記のようにB2Gマニフェストを取得します:

    git clone https://github.com/mozilla-b2g/b2g-manifest.git

    リポジトリ内のnexus-s.xmlファイルを修正し、linaro gitエントリーをasopエントリーに置き換え、下記のようにします:

    <default revision="refs/tags/android-4.0.4_r1.2"
              remote="aosp"
              sync-j="4" />

    変更をコミットし (git commit -a) 、次に、チェックアウトしたB2Gリポジトリのマスターブランチ内のconfig.shファイルを編集し、あなたが修正したMozillaのマニフェストの代わりにローカルのマニフェストを指示するようにします:

    GITREPO=${GITREPO:-"file:///home/path/to/my/b2g-manifest"}

    MacのXcode 5を使ってビルドしている時のclangのエラー

    Mac OS X 10.8上のXcode 5でビルドしている時、下記の様なエラーをよく見かけるでしょう:

    clang: error: argument unused during compilation: '-include system/core/include/arch/darwin-x86/AndroidConfig.h'
clang: error: argument unused during compilation: '-U DEBUG'
clang: error: argument unused during compilation: '-U DEBUG'
clang: error: argument unused during compilation: '-MF out/host/darwin-x86/obj/EXECUTABLES/obbtool_intermediates/Main.d'
make: *** [out/host/darwin-x86/obj/EXECUTABLES/obbtool_intermediates/Main.o] Error 1

    これはXcode 5がg++ コンパイラを/usr/binに変更しているのが原因で、コンパイル時にそれを使おうとする時にビルドプロセスが停止します。この問題を回避するには build/core/combo/HOST_darwin-x86.mkの下記の行を編集して下さい:

    HOST_CXX := g++

    から

    HOST_CXX := g++-4.6
ifeq (,$(wildcard /usr/local/bin/g++-4.6))
 HOST_CXX := g++
endif

    まで。次にgccをアンインストールするため、brewを使います(すでにMac OS bootstrap script を使っている場合 — そうでない場合はそれを終わらせておく必要があります):

    brew uninstall gcc-4.6

    ここでc++ をサポートしたgccとmultilibを再インストールします:

    brew install --enable-cxx https://gist.github.com/artlogic/6988658/raw/aeb9d1ea098274ad3f3fe2637b9df7f308a8a120/gcc-4.6.rb

    PATH に /usr/local/bin iの記載があることを確認して下さい。一時的には、コマンドプロンプトで下記のようにタイプすることで可能です:

    export PATH=/usr/local/bin:$PATH

    上記をホームディレクトリの.bash_profileに追加しておけば、この変更をずっと使えます。

    PATHの設定後、下記コマンドを両方実行してください:

    gcc-4.6 -v

g++-4.6 -v

    コマンドが失敗する場合、下記コマンドのようにbrewでgccを再リンクします:

    brew link --overwrite gcc-4.6

    /usr/bin/c++ がXcode 5インストール時のように clang++ を指してなくなっている場合、下記も可能です。下記のようにタイプして決めることができます:

    ls -l /usr/bin/c++

    このようなのが返ってくるはずです:

    lrwxr-xr-x 1 root admin 7 Sep 19 11:40 /usr/bin/c++ -> clang++

    c++ iが clang++以外を指してる場合、下記コマンドで更新します:

    sudo rm /usr/bin/c++

sudo ln -s /usr/bin/clang++ /usr/bin/c++

    Cannot pull files from backup directory(backupディレクトリからファイルを取得できない)

    これは(バックアップ)デバイスからコンピュータにデータ取得している時に、USB接続が壊れた場合に起こります。

    もう一度スクリプトを実行すると、おそらく下記のような出力(これはPeakデバイスの例)が出るでしょう:

    <code>Pulling files from ../../../backup-peak
cat: ../../../backup-peak/system/build.prop: No such file or directory
Found firmware with build ID
Pulling "libaudioeq.so"
cp: cannot stat `../../../backup-peak/system/lib/libaudioeq.so': No such file or directory
Failed to pull libaudioeq.so. Giving up.

> Build failed! <

Build with |./build.sh -j1| for better messages
If all else fails, use |rm -rf objdir-gecko| to clobber gecko and |rm -rf out| to clobber everything else.</code>

    これを解決するには、objdir-geckoや出力ディレクトリの全体を削除する必要はありません。バックアップのディレクトリだけを、このように消します(上記サンプルむけ):

    <code class="brush: bash">$rm -rf backup-peak</code>

    エミュレータのビルドの問題

    emulator をビルドする時は、下記の問題にご注意ください：

    まず、x86のエミュレータは使用すべきではありませんーーーインストール困難で、サポートが良くないです。

    次に、エミュレータ用のビルドシステムは、32bit/64bit両方のエミュレータをビルドします。エミュレータはOpenGLに依存しているるため、32bit/64bit両方のOpenGLライブラリをシステムにインストールする必要があります。bug 897727(英語)の議論を見てください。

    この問題を解決するには２つの方法があります：

    解No.1: 32bitと64bit両方のOpenGL libsをインストールし、正しくsymlinksを保つ

    もし使用するlinuxディストリビューションにOpenGLライブラリのmultilibパッケージがあるなら、インストールを試すことができます。そして手動でシンボリックリンクを作ることになるでしょう。

    例えば、Ubuntu 12.04 LTS x86-64での状況を示します。このディストリビューションではlibgl1-mesa-devパッケージをx86-64版とi386版を同時にインストールできません。しかし、下記組み合わせのパッケージを同時インストールできます:

    sudo apt-get install libgl1-mesa-dev libglapi-mesa:i386 libgl1-mesa-glx:i386

    このコマンドを実行後、エミュレータのビルドを成功させるには、まだ手動でシンボリックリンクを作る必要があります。:

    $ sudo ln -s /usr/lib/i386-linux-gnu/libX11.so.6 /usr/lib/i386-linux-gnu/libX11.so
$ sudo ln -s /usr/lib/i386-linux-gnu/mesa/libGL.so.1 /usr/lib/i386-linux-gnu/libGL.so

    解No.2: エミュレータが64bitだけをビルドするようパッチを当てる

    単に このパッチ をB2Gリポジトリ以下のsdk/ git リポジトリに適用しまうｓ．これは64bitシステム上ではB2Gエミュレータが64bitのエミュレーターだけをビルドするようにします。 ゆえにmultilibのいかなる問題も回避します。64bitシステムではいずれにせよ32bitエミュレータを使用しません。これは、このパッチが最 終的にゴミクズ化（原語:bit-rot）するまでは最もシンプルな解決策です。

    B2G/Firefox OS/Gaiaのバグ登録

    いったんB2G/Firefox OSのビルドを終えて実行すると、特定バージョンのバグを記録したくなるでしょう。Firefox OSコミュニティでは出来るだけ効率的に改善ができます。

    "Firefox OS"の下でBugzillaのバグ登録をしたくなるでしょう。ただバージョン情報を含める必要があります:

    1. まず最初に、バージョン番号、例えば1.4.0.0-prereleaseを教えてください。端末上ではSettings > Device Informationにあります。
    2. gaiaとgeckoリポジトリから現在のハッシュを取得することで、もっと詳細なバージョンidを報告できます。このようにして: 
      #!/bin/bash
(cd gaia;  echo "gaia  $(git rev-parse HEAD)")
(cd gecko; echo "gecko $(git rev-parse HEAD)")G
