    Firefox OS を入手してビルドする前に、適切に設定されたビルド環境（このページで方法を説明する）が必要です。現在 64 ビット版 Linux ディストリビューション もしくは Mac OS X でビルド可能です。

    注意：　ビルドしたFirefox OSをスマートフォンに焼くには、最初からコンピューターに接続「しないで」ください。いつ挿すべきかは後述します。

    互換性のあるスマートフォンを用意するか、エミュレータを使用する

    ビ ルドするには互換性のある端末を用意する必要があります。さもなくばエミュレータを使用してください。我々は数種類のスマートフォンをサポートしてはいま すが、複数モデルで利用可能なものと、限られたモデルでしかサポートされていないものがあります。その上、他のものよりも良くサポートされているデバイス もあります。現時点では以下のデバイスで Firefox OS の移植版が利用可能です :

    記: Mozillaの B2Gソースコードは B2Gの効率的なリファレンス実装ですが、電話ベンダーは自由にパッチをあてたり変更することが可能です。例えばAlcatel One Touchを買った時、ベンダーのバージョンを含みます。これはインストールされるアプリには影響はありませんが、プラットフォームレベルでは影響する可能性があります。Nexus 4の移植はMozillaがメンテしています、なので他のティア３の端末と比較して、はるかにうまく我々のGeckoのバージョンと連動する可能性が高く なる傾向にあります。

    ティア 1

    ティア 1 の端末は開発の主要なターゲットとして積極的にサポートされているものです。大抵の機能を利用できるようになるのも、バグ修正されるのも、最初に着手されるのはこれらのデバイスです。

    Keon
    KeonはGeeksphoneのKeon端末です。これは初期の開発機の一つです。この端末のビルドはGeeksphoneから提供されることにご注意ください。
    Inari
    Inariはその他のテスト端末です。この設定をZTE Openの実機ビルドに使用してください。
    警告: Firefox OSの新し目のビルドは、ZTE Openのデフォルトビルドパーティションからの起動にトラブルをもたらす可能性があります。
    Flame
    FlameはMozillaの新しい標準のFirefox OSリファレンス機です。よってZTE Open C用のビルド時にもFlameのコンフィグレーションを使うべきです、なぜならFlameとOpen CはAndroid Jellybeanをベースにしているため。
    エミュレータ（ ARM ・ x86 ）
    利用可能なエミュレータは 2 種類あります。一つは ARM コードをエミュレートし、もう一つは 実行するのに x86 コードを使用します。「B2Gエミュレータを使用する」で詳しく学んでください。
    x86のエミュレータは使用すべきでないことにご注意ください。－インストールが困難で、十分にサポートされていません。 
    デスクトップ
    デスクトップ版の Firefox OS をビルドすることもできます。このビルドでは、XULRunner アプリケーションで Gecko を実行し、その中で Gaia のユーザ体験を確認できます。
    Flatfish
    Flatfish is the first tablet program of Firefox OS. Even though some of functionalities about telephony won't work on flatfish. You'll need do some extra settings to be able to build code.

    もちろん、スマートフォンがなくても、デスクトップクライアントや、エミュレータをビルドすることは可能です。

    ティア 2

    ティア 2 のデバイスは大抵の機能が動作し、多くの開発者（特にアプリの開発者）が使用しているものですので、二番目に優先対応される環境です。

    Samsung Nexus S
    Nexus S の動作確認済みモデルは GT-I9020A と GT-I9023 です。他のモデルでも動作するかもしれません。
    Samsung Nexus S 4G
    SPH-D720 がティア 2 のデバイスとしてサポートされています。

    ティア 3

    これらのデバイス用の Firefox OS はビルド可能ですが、コア開発者によって定期的に活発な開発が行われているわけではありません。そのため、信頼性や機能セットは、ティア 1 やティア 2 デバイスよりも後になり、かなり遅れたものになるでしょう。

    Samsung Galaxy S2
    動作するモデルは i9100 のみです。他に公式的に互換性のあるモデルはありません。 (i9100P は NFC チップ追加の変更だけなので動作する可能性があります)
    Samsung Galaxy Nexus
    現在どのモデルが互換性がないのかわかりません。
    Nexus 4
    IRC上の数名がテストして成功しています。Android4.4が動作している場合、最初にAndroid 4.3に焼き直しする必要があったりなかったりします(Android images available from Google)。
    Nexus 5
    IRC上の数名がテストして成功しています。
    Tara
    Taraはもう一つの開発端末です。Taraのマニフェストはマスターブランチのみです。Taraのコードを入手するスクリプトは"BRANCH=master ./config.sh tara"です。

    Unagi

    Unagi は下位から中位機種のスマートフォン向けのテスト及び開発プラットフォームとして使用されているスマートフォンです。Firefox OS の主な開発者の多くは Unagi で作業しています。
    Pandaboard
    Pandaboard は OMAP 4 アーキテクチャベースの開発用基板で、モバイルプラットフォームの開発作業を行うのに使用されています。
    重要 : Android 4 (いわゆるIce Cream Sandwich) 以降で動作しているデバイスのみがサポートされています。もしあなたのデバイスが上のリストにあっても、古いバージョンの Android を実行している場合は、最初に更新してください。

    Flame and Nexus 5 libxml2-utils requirement

    If you are building for the Flame reference device or Nexus 5, you should first run the following command in Terminal:

    sudo apt-get install libxml2-utils

    ソフトウェア、ハードウェアホームボタンとホームジェスチャー

    すべてのティア１端末にはホームボタンがあり、ユーザーをホームスクリーンに戻す事ができます。たいていの現在のICSベースのAndroid端末は、ナビゲーションにオンスクリーンのタッチボタンを使用しています。今はハードウェアホームボタンがなくても、仮想ホームボタンがあります。それが自動的に有効にならない場合、設定アプリを開き、Developer settingsでEnable software home button の設定値を変更してください。

    ティア２とティア3の端末にはハードウェアウェアホームボタンの代わりにソフトウェアホームボタンがあります。

    Firefox OS1.4では"Home gesture enabled"の開発者オプションがあります。つまりそのオプションではスクリーン上のホームボタンが削除され、スクリーン下端を上にスワイプする方が選ばれます。

    GNU/Linux での必要条件

    Linux でビルドするには以下のシステム構成が必要です :

    • インストール済みの 64 ビット版 GNU/Linux ディストリビューション （ Ubuntu 12.04 推奨 ）
    • 最低 4 GB の RAM 及び スワップ空間　//訳注:2GBのVPSで実績あり
    • 最低 20 GB のハードディスク空間

    これは最小限度よりは多いのですが、リソース不足でビルドに失敗することがあります。その場合の典型的なエラーは"arm-linux-androideabi-g++: Internal error: Killed (program cc1plus)"です。

    以下のツールもインストールしておく必要があります :

    • autoconf 2.13
    • bison
    • bzip2
    • ccache
    • curl
    • flex
    • gawk
    • git
    • gcc / g++ / g++-multilib
    • make
    • OpenGL shared libraries
    • patch
    • X11 headers
    • 32-bit ncurses
    • 32-bit zlib

    64 ビット版のインストール例 :

    このセクションでは、FirefoxOSをビルドするのに必要なもの全てをインストールために、異なるLinuxディストリビューションで実行するコマンドを列挙します。
    //訳注：ディストリビューションによってドキュメント変更が激しいので、英語版ドキュメントも見るようにしてください

    Ubuntu 12.04 / Linux Mint 13 / Debian 6

    ターミナルで下記のコマンドを実行します：

    $ sudo apt-get install autoconf2.13 bison bzip2 ccache curl flex gawk gcc g++ g++-multilib git ia32-libs lib32ncurses5-dev lib32z1-dev libgl1-mesa-dev libx11-dev make

    リファレンス機の"Flame" またはNexus 5用にビルドするには、ターミナルで下記コマンドを実行してください:

    sudo apt-get install libxml2-utils

    そして上記のエミュレータでのビルドの問題を見てください！
     

    Ubuntu 12.10 / Debian 7

    ターミナルで下記のコマンドを実行します：

    $ sudo apt-get install autoconf2.13 bison bzip2 ccache curl flex gawk gcc-4.6 g++-4.6 g++-4.6-multilib git ia32-libs lib32ncurses5-dev lib32z1-dev libgl1-mesa-dev libx11-dev make

    上記のエミュレータのビルドの問題に加え、コンパイラが gcc-4.7をデフォルト指定するため、下記のエラーが出てビルドに失敗します:

    "KeyedVector.h:193:31: error: indexOfKey was not declared in this scope, and no declarations were found by argument-dependent lookup at the point of instantiation"

    これを修正するため、デフォルトホストコンパイラを GCC 4.6 に指定する必要がありますので、B2Gのソースの取得後に、デフォルトホストコンパイラの変更方法を参照してください。

    新しくUbuntu 12.10をインストールした時、ia32-libsに関するunmetの依存性エラーが出るでしょう。 下記コマンドにより修正します：

    sudo dpkg --add-architecture i386
sudo apt-get update
sudo apt-get install ia32-libs

    Ubuntu 13.04

    ターミナルで下記のコマンドを実行します： 

    $ sudo apt-get install --no-install-recommends autoconf2.13 bison bzip2 ccache curl flex gawk gcc g++ g++-multilib gcc-4.6 g++-4.6 g++-4.6-multilib git ia32-libs lib32ncurses5-dev lib32z1-dev zlib1g:amd64 zlib1g-dev:amd64 zlib1g:i386 zlib1g-dev:i386 libgl1-mesa-dev libx11-dev make zip

    上記のエミュレータのビルドの問題に加え、コンパイラが gcc-4.7をデフォルト指定するため、下記のエラーが出てビルドに失敗します:

    "KeyedVector.h:193:31: error: indexOfKey was not declared in this scope, and no declarations were found by argument-dependent lookup at the point of instantiation"

    これを修正するため、デフォルトホストコンパイラを GCC 4.6 に指定する必要がありますので、B2Gのソースの取得後に、デフォルトホストコンパイラの変更方法を参照してください。

    Ubuntu 13.10

    Ubuntu 13.10では、マルチアーキパッケージは、マルチなアーキテクチャ (つまり32-bitを64-bitOS上にインストール)の主流な方法です。Ubuntuシステムに対し、32-bitパッケージも同様にサポートしてほしいと通知する必要があります:

    sudo dpkg --add-architecture i386
sudo apt-get update

    一度それを終えると、必要なパッケージをインストールできるようになります:

    sudo apt-get install --no-install-recommends autoconf2.13 bison bzip2 ccache curl flex gawk gcc g++ g++-multilib gcc-4.6 g++-4.6 g++-4.6-multilib git lib32ncurses5-dev lib32z1-dev zlib1g:amd64 zlib1g-dev:amd64 zlib1g:i386 zlib1g-dev:i386 libgl1-mesa-dev libx11-dev make zip libxml2-utils 

sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-4.6 1 

sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-4.8 2 

sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-4.6 1 

sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-4.8 2 

sudo update-alternatives --set gcc "/usr/bin/gcc-4.6" 

sudo update-alternatives --set g++ "/usr/bin/g++-4.6"

    Ubuntu 14.04

    Ubuntu 13.10のマニュアルに従う。

    Fedora 19/20

    ターミナルで下記のコマンドを実行します：

    $ sudo yum install autoconf213 bison bzip2 ccache curl flex gawk gcc-c++ git glibc-devel glibc-static libstdc++-static libX11-devel make mesa-libGL-devel ncurses-devel patch zlib-devel ncurses-devel.i686 readline-devel.i686 zlib-devel.i686 libX11-devel.i686 mesa-libGL-devel.i686 glibc-devel.i686 libstdc++.i686 libXrandr.i686 zip perl-Digest-SHA wget

    Fedoraに付属のデフォルトコンパイラでもプロジェクトを適切にビルドできるでしょうが、 コンパイルに問題がある場合、GCC 4.6.xインストールする必要があるかもしれません:

    • Pre-compiled GCC 4.6.x for Fedora 19/20

    使用するFedoraに適合したバージョンをダウンロードし、/optにインストールします。Fedora 19/20には下記のコマンドを:

    $ curl -O http://people.mozilla.org/~gsvelto/gcc-4.6.4-fc19.tar.xz
$ sudo tar -x -a -C /opt -f gcc-4.6.4-fc19.tar.xz

    デフォルトホストコンパイラを GCC 4.6 に指定する場合、B2Gのソースの取得後に明示的にセットする必要がありますので、デフォルトホストコンパイラの変更方法を参照してください。

    コンパイラがlibmpc.so.2が見つからないと文句を言ってビルドに失敗する場合、compat-libmpcパッケージを入れます。

    Arch Linux

    ターミナルで下記のコマンドを実行します：

    $ sudo pacman -S --needed alsa-lib autoconf2.13 bison ccache curl firefox flex gcc-multilib git gperf libnotify libxt libx11 mesa multilib-devel wget wireless_tools yasm zip lib32-mesa lib32-mesa-libgl lib32-ncurses lib32-readline lib32-zlib

    lib32-*パッケージをインストールするには、 multilibリポジトリを有効にしておく必要があります。
    B2Gはgcc4.6.4でのみコンパイル可能であり、 またArch Linuxは常に先鋭的なソフトウェアを同梱しているため、gcc46-multilib from AURをインストールする必要があります。 気をつける点として、PKGBUILDを編集して options 列にand add staticlibs を追加しないといけません。そうしないとgccはB2Gのコンパイルに失敗し、 コンパイル時にcannot find -lgcc のエラーを吐きます。下記の行を.userconfig ファイルに加える必要もあります:

    export CC=gcc-4.6.4
export CXX=g++-4.6.4

    初期状態では、Arch Linux はPython3を使用します。強制的に古いpython2を使うようにする必要があります。python2の実行モジュールをpythonにリンクする こともできますが、お勧めできませんし、過ちの元になると考えられます。これはインストールされているpython3を破壊します。より良い方法はvirtualenv/virtualenvwrapperを使うことです:

    $ sudo pacman -S python-virtualenvwrapper
$ source /usr/bin/virtualenvwrapper.sh
$ mkvirtualenv -p `which python2` firefoxos
$ workon firefoxos

    Androidはmake 4.0の代わりにmake 3.81 or make 3.82 を使えと文句を言います。  AURからmake 3.81をダ ウンロードできます。これは make-3.81 バイナリをあなたのパスにインストールし、ビルド時に正しいバージョンを使うために、makeと名付けられたシンボリックリンクを環境変数PATHの前に作成する必要があります。

    mkdir -p ~/bin
ln -s `which make-3.81` ~/bin/make
export PATH=~/bin:$PATH

    このほかAndroidはJava6 SDKが必要ですが、Archには Java7だけがあります。 AURのビルドが不幸にも壊れていますが、手動で Java 6 SDK をダウンロード、インストールすることができます。それをパス上に置く必要があります。

    cp ~/Downloads/jdk-6u45-linux-x64.bin /opt
su
cd /opt
chmod +x jdk-6u45-linux-x64.bin
./jdk-6u45-linux-x64.bin
exit
ln -s /opt/jdk1.6.0_45/bin/java ~/bin/java

    Gentoo Linux

    ccacheのインストール

    ccacheをインストールする必要があります。これは部分的なビルドをキャッシュしてくれます。

    # emerge -av ccache

    ccacheはサポート問題を良く起こすとわかっているので、Gentooは明示的に、控えめに使うのをおすすめします。

    ./build.sh スクリプトが呼ばれる後のステップでccacheの使用が必要になる時、Gentooユーザーは代わりに明示的に展開されたパスと共に実行します。つまりこうなります。

    $ PATH=/usr/lib64/ccache/bin:$PATH ./build.sh
    部分イメージの生成

    B2Gを実際の物理的ハードウェア用にビルドする場合、色々な点から、一部分のイメージを端末にアップロードしたくなるかもしれません。 (例えば、fastbootユーティリティを使ってバックアップファイルを復旧する時)

    ここではファイルシステムイメージがYAFFS2 (Yet Another Filesystem 2)とします。 Gentooは最新の(ie. git HEAD) yaffs2-utils ユーザーランドのパッケージをサポートします。 (記: YAFFS2イメージをマウントする場合カーメルパッチも必要になります。 ですが、これは真に必要とは言えません、代わりに分解してリビルドすることが可能なため。)

    # emerge --autounmask-write yaffs2-utils; etc-update; emerge -av yaffs2-utils

    こうしたイメージを生成するには、単にパッケージ化する部分ファイルシステム木の親ディレクトリに移動し、このようなコマンドを発行します:

    $ mkyaffs2image system/ system.img

    Mac OS X での必要条件

    Mac OS X上でFirefox OSをビルドするには、下記に示すような、数々の前提となる手順に従う必要があります。特定の状況で起こりうるよくあるエラーや、その解決法についても議論します。

    記: B2GをKeon用にコンフィグ・ビルドすることはMacで動作しません。 この端末用にB2GをビルドするにはLinuxを使う必要があるでしょう。

    バージョン互換性

    XCode 4.2以前は、Mac OS X 10.9 (つまり"Mavericks")と互換性がありません。なのでFirefox OSのビルドが失敗することに驚くかもしれません。基本的に、Mac OS X 10.9以降では、もはやICSのエミュレータもビルドできず、flatfishはXCode 5.xシリーズでのビルドに失敗します。

    Mac OS X 10.9.xでのバージョン互換性
      emulator (ICS) flatfish/vixen (JB-4.2) emulator-jb (JB-4.3) emulator-kk (KitKat-4.4)
    XCode 4.3.3 X X X X
    XCode 4.4.1 X O[1] O O
    XCode 4.5.2 X O[1] O O
    XCode 4.6.3 X O[1] O O
    XCode 5.0.2 X X O O
    XCode 5.1.1 X X O O
    1. flatfish/vixenをビルドするには環境変数を BUILD_MAC_SDK_EXPERIMENTAL=1 と設定・エクスポートする必要があります。

    XCode Command Line Utilitiesをインストールする

    XcodeのCommand Line Utilitiesをインストールする必要があります。 Apple's developer downloads page からOS Xのバージョン用のCommand Line Utilitiesをダウンロードできます。しかしながらXCodeのアプリケーションスイート全体が良い場合、Mac App Storeを通じて Xcodeをインストール できます。

    Xcode 4.3.1 (OS X 10.7 "Lion")　及び 4.4.1+ ( Mac OS X10.8 "Mountain Lion")等のより新しいバージョンでは、必要となる Command Line Tools が必ず入っているわけではありません。XCode をインストールする際に、必ず Preferences の Downloads パネルで Command Line Tools をインストールすることを忘れないでください。さらに、最低でも空きディスク容量が 20GB あるかどうか確認してください。

    記 : Firefox OS エミュレータには、Core 2 Duo プロセッサ以降、Mac OS X 10.7 “Lion” と互換性のあるシステムが必要です。実際に Lion が動作している必要はありませんが、互換性は必要です。まあとにかく、多くの古めのMac 上で Firefox OS のあらゆるビルドが可能です。

    記: XCode 4.2.x以前はMac OS X 10.9 以降と互換性がなく、XCode 4.3.x は10.8用のプラットフォームSDKがありません。より新しいバージョンをインストールしてください。またflatfishについての作業では、XCode 4.4, 4.5, 4.6のいずれかが使用できることを確認してください。

    Firefox OS Mac ブートブートストラップを実行する

    curl -fsSL https://raw.github.com/mozilla-b2g/B2G/master/scripts/bootstrap-mac.sh | bash

    それから、ターミナルを開き、上記のコマンドをターミナルで実行する必要があります。これはブートストラップスクリプトのプルと実行をして、全ての必要条件を満たしているかを確認するものです。実行すると、エミュレータのビルドを可能にするのに必要となる全てのものが揃っているかどうか調べ、不足しているものがあれば、インストール許可確認用のプロンプトを出したり、警告を出したり、問題の解決案を示唆します。このスクリプトが確認とインストールを行う項目は次の通りです :

    • git
    • gpg
    • ccache
    • yasm
    • autoconf-213
    • gcc-4.6
    • gnu-tar
    • homebrew

    Xcodeの強烈な議論の例

    もし、Xcode 4.4+ に既にアップグレードしていて、Xcode が無効であるというメッセージを受け取った場合、Xcode のパスを確認してください :

    xcode-select -print-path
    /Developer を指したままになっている時は、パスを更新してください : 
    sudo xcode-select -switch /Applications/Xcode.app

    Mac OS X 10.6 SDKを利用可能にする

    次に Mac OS X 10.6 SDK が利用可能かどうか確認します : 
    /Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/

    もしそこになかったら、XCode 4.3.を展開してコピーする必要があります。

    1. Apple Developer ポータルで入手可能な Xcode 4.3 DMG ファイルからダウンロードします(Apple Developer accountが必要になります)。
    2. Pacifist ユーティリティをダウンロードし、XCode 4.3 .dmgファイルから10.6 SDK を展開するのに使用します。"Open Package"ボタンをクリックし、検索窓に10.6と入れてSDKを検索し、次にMacOSX10.6.sdkディレクトリを選んで"Extract To..."ポタンで適切な場所に展開します。
    3. 10.6 SDK の場所から /Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/ ディレクトリへのシンボリックリンクを追加してください。例えば、10.6 SDK をデスクトップ上に置いている場合、コマンドは下記のようになります。
    ln -s /Users/<yourusername>/Desktop/MacOSX10.6.sdk /Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/

    記: これはMac OS X 10.9以降では不要です。なぜなら10.6 SDK はICS用で、ICSはMac OS X 10.9をサポートしないため。

    Macのファイルシステムの大文字小文字区分について意識しておく

    Mac OS X はデフォルトでは大文字と小文字の区別をしないファイルシステムです。 Linux カーネルには同一で大文字小文字の異なるファイルが多数あるため、これが問題となります。  例えば、ヘッダーファイルの xt_CONNMARK.h と xt_connmark.h。その結果、/kernel の中で多数のファイルが./config.shの後に修正されます。

    多くの場合、ビルドはうまくいきます。いくつかのプラットフォームでは、しかしながら、下記のエラーに出くわす可能性があります:

    ERROR: You have uncommited changes in kernel
You may force overwriting these changes
with |source build/envsetup.sh force|

ERROR: Patching of kernel/ failed.

    bug 867259 にて、多くの議論と、問題の修正策を見て下さい。

    他には、大文字と小文字を区別するファイルシステム上でビルドするのが、常に最も安全でしょう。その最も簡単な方法は、分離され、マウント可能な、大文字と小文字を区別するディスクイメージを作成することです。Apple製のディスクユーティリティを使うことで可能ですし、またはコマンドラインでは :

    hdiutil create -volname 'firefoxos' -type SPARSE -fs 'Case-sensitive Journaled HFS+' -size 40g ~/firefoxos.sparseimage

    ドライブをマウントするには:

    open ~/firefoxos.sparseimage

    マウントされたドライブに移動するには:

    cd /Volumes/firefoxos/

    次にこの場所において、コードをチェックアウトし、大文字小文字問題を心配することなくコンパイルできます。

    Mountain Lion homebrew gotcha

    Mountain Lion で作業していて、homebrewを通じて依存するパッケージのインストール中に、以下のようなエラーが発生した場合 :
    clang: error: unable to execute command: Segmentation fault: 11
    --use-gcc フラグを追加して、手動でエラーになった依存パッケージを再インストールしてみてください・・・。例としては : 
    brew install mpfr --use-gcc

    Samsung Galaxy S2用の追加手順

    もし Samsung Galaxy S2 用にビルドする予定なら、heimdall もインストールする必要があります。 詳細は heimdall のインストール を参照してください。ブートストラップスクリプトは実行しません！

    注意 : もし多くの Samsung 製スマートフォンの内容管理に使用される Samsung Kies ツールをインストール済みなら、デバイス上で Gecko をフラッシュメモリから起動させる前に削除する必要があります。Windows では、標準のアプリケーション削除処理を使用できます。Mac では Kies インストールディスクイメージに、システムから完全に Kies を削除するユーティリティがあります。 Kies がインストールされたままだと、フラッシュメモリへの書き込みがうまく動作しないでしょう。もし Kies を削除することを忘れても、ビルドシステムはそれを検知して、アンインストールするよう知らせてくれます。アンインストールツールが正しく ~/Library/Application Support/.FUS フォルダーを削除できなかった場合も、ユーザースタートアップ項目リストにあるユーティリティの参照を中止しますので、注意してください。それらは手動で削除することになるでしょう。

    libmpc の依存関係が破損していた場合の修正方法

    gcc 4.6 は libmpc 0.9 でビルドされていますが、homebrew を使用してパッケージを更新した場合、libmpc がバージョン 1.0 に更新されても、homebrew はライブラリのバージョン変更後に gcc 4.6 をリビルドしません。その場合、再び機能するようにするには、こんな感じにシンボリックリンクを作成する必要があります :

    cd /usr/local/lib/
ln -s libmpc.3.dylib libmpc.2.dylib

    任意 : HAX のインストール

    Intel は、x86 エミュレータを使用している人に対して、Mac 上で、B2G エミュレータがエミュレートされたコードではなくネイティブコードで動作させられる特別なドライバーを提供しています。これを使用したい場合はダウンロードしてインストール可能です。必ず必要なものではありませんが、エミュレーションのパフォーマンスと安定性が向上します。

    HAXのインストール前に Android SDKを入れる必要があるでしょう。

    adb のインストール

    ビルドプロセスは、B2G をビルドする前に（もちろん、エミュレータをビルドする場合を除いて）、スマートフォンにインストールされた Android からバイナリ BLOB をプルする必要があります。このため adb (Android Debug Bridge) が必要になります。ADBをインストールして使用するの記事でインストール方法を説明しています。

    Note for future when you start to use adb: adb needs the phone's lock screen to be unlocked in order to see your phone (at least in later versions of Firefox OS). You'll probably want to disable the lock screen (we'll get to how later in the build instructions).

    heimdall のインストール

    heimdall は Samsung Galaxy S2 のフラッシュメモリに書き込みするためのユーティリティです。Boot to Gecko フラッシュユーティリティが、Firefox OS がインストールされたスマートフォンの内容を置き換える場合や、デバイス上の B2G と Gaia の更新バージョンの書き込みの際に使用されます。Galaxy S2 上に Firefox OS をインストールしたい場合は必要ですが、その他のデバイスでは必要ありません。他のデバイス向けには、代わりにfastbootユーティリティをビルドして使用しています。

    注意 : 繰り返しですが、これは Samsung Galaxy S2 に Firefox OS をインストールする場合にのみ必要であることに注意してください。

    heimdall をインストールするには 2 通りのやり方があります :

    • GitHub からコードをダウンロードして、自分でビルドする。
    • パッケージマネージャを使用してインストールする。
      • Linux　: sudo apt-get install libusb-1.0-0 libusb-1.0-0-dev
      • Mac では、インストール用パッケージをダウンロードしてそれを使用することができます。

    ccache の設定

     ccache --max-size 10GB

    Linux用：あなたのスマートフォンに対する udev ルールの設定

    記: この章はLinux特有です。Mac OS Xでは必要となる端末の権限がセットアップ済みになっています。

    次に、端末用のudevルールを設定します。

    電話機を接続した状態でlsusb を実行すると、USBベンダーID を取得できますが、通常は Google の18d1、Samsung の04e8、ZTEの19d2、Geeksphone/Qualcommの05c6になります。/etc/udev/rules.d/android.rules ファイルに以下の行を追加します(XXXXをあなたの端末のIDに置き換える):

    SUBSYSTEM=="usb", ATTR{idVendor}=="XXXX", MODE="0666", GROUP="plugdev"

    ZTEを例に取ると、android.rulesの内容はこうなります。

    SUBSYSTEM=="usb", ATTR{idVendor}=="19d2", MODE="0666", GROUP="plugdev"

    Firefox Flameではこのように:

    SUBSYSTEM=="usb", ATTR{idVendor}=="05c6", MODE="0666", GROUP="plugdev"

    fastboot device(flash時に使います)では異なるベンダーIDを持つ事もありますし、そのためのルールが必要になります。

    Flameの例では GoogleのベンダーIDも要ります。

    SUBSYSTEM=="usb", ATTR{idVendor}=="18d1", MODE="0666", GROUP="plugdev"

    ファイルが存在しない場合、作成します。rules.d ディレクトリは通常はデフォルトでリードオンリーになっています。なのでchmodを使ってディレクトリ、ファイル、または両方を書き込み可能にしないといけない場合もあります。

    一旦このファイルを保存して終了し、ファイルに読み込み属性を付けます :

    sudo chmod a+r /etc/udev/rules.d/android.rules

    今やudevルールが更新されたので、udevデーモンを再起動します。

    Ubuntu

    sudo service udev restart

    Fedora 20

    sudo systemctl restart systemd-udevd.service

    最終的に、USBケーブルを抜いて接続しないようにします。何故なら最初に電話機上でリモートデバッグを有効にする必要があるため。

    リモートデバッグを有効にする

    USB ポートにスマートフォンを接続する前に、USB 開発者モードにしてください。デバッグ及びフラッシュメモリの更新で、スマートフォンにアクセス可能になります。これを行うには、端末で Developer settingsのRemote Debugging（以前は開発者モードと呼ばれていました）を有効にしてください。オプションがチェックされると、リモートデバッグが有効になり、準備が整います。

    この点において、USBケーブル経由で電話機とコンピューターを接続します(以前にudevルールを作成していれば、これを起点にudevが電話機を検出して、適切なパーミッションのもと端末ノードを作成します)。いまやadb devicesコマンドにより端末リストを出せるかどうかチェックできます(adbはスクリーンロックが解除されている時だけ電話機を見られることを忘れないで下さい)。全てがうまくいった場合、これに似た出力が見えるでしょう(下記はGeeksphone Keon用のもの):

    $ adb devices
List of devices attached
full_keon       device

    端末リストが期待と異なる場合、ファイル名やスクリプトが全て正しいかチェックします(前節を見よ)。その後コンピューターを再起動して再度コマンドをタイプします。また端末がfastbootを使っている時、ブートローダーは自身を通常起動時に見えるベンダーIDと識別することがあります。

    スマホのシステムパーティションをバックアップする

    記: システムバックアップを取得していない場合、端末をビルドする前にこれを行う必要があります。何故ならビルド時にいくつかのライブラリが参照されるためです。これらのライブラリはプロプライエタリな場合があり、このためコードベース内で提供できません。

    電話機の全てのAndroidシステムパーティションをバックアップすることが推奨されます。

    あとでB2Gツリーを削除する場合、Androidのバイナリーブロブのコピーを使用できます。そうするには、こう実行します:

    adb pull /system <backup target dir>/system

     電話機によっては、この他に/dataや/vendor といったディレクトリをプルする必要があります:

    adb pull /data <backup target dir>/data
adb pull /vendor <backup target dir>/vendor

    "insufficient permission（不十分なパーミッション）"メッセージでプルに失敗する場合、下記を試します:

    • adbサーバーを停止、再起動します、あるいはそれでうまくいかないなら
    • カスタムROM内のadbツールにルート権限が与えられているか再チェックします(e.g. under CyanogenMod, change Settings > System > Developer Options > Root Access to Apps and ADB or ADB only).
    • udevルールが正しくセットアップされているか検証します(see For Linux: configure the udev rule for your phone.

    次の段階へ

    この時点で、Firefox OS のコードを取得する準備ができているはずです !
