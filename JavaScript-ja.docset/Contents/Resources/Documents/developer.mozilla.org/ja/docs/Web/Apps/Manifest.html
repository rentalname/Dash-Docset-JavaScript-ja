    Open Web App の マニフェスト には、Web ブラウザがアプリとやりとりするのに必要な情報が含まれます。マニフェストは、Open Web App を Web サイトと区別する重要なポイントのひとつです。これはアプリの名称と説明を記載した JSON ファイルで、アプリの配信元、アイコン、そのアプリに必要な許可設定などを含めることもできます。マニフェストを扱うブラウザは Web ランタイム を組み込む必要があります。この記事ではマニフェストに含めることのできる内容を詳細に解説します。

    マニフェストファイルには manifest.webapp というファイル名を付け、index.html ファイルとともに Web アプリディレクトリのルートに配置する必要があります。

    自分が管理するページからアプリを自主配布する場合は、(例えばボタンから navigator.mozApps.install() を呼び出すことで) アプリのインストールを実行します。ストアやマーケットプレイスでアプリを配布する場合は、そのストアが掲載アプリのマニフェストの URL をブラウザに提供することでインストールを実行します。

    アプリマニフェストに関するよくある質問 も参照してください。マニフェストバリデータ も利用できます。このツールを使えば、マニフェストの内容を検証し、エラーを事前に特定できます。

    マニフェストの項目

    マニフェストには以下の項目を含めることができます。マニフェスト検証時の必須項目は name、description、(128px のアイコンを含む) icons の 3 つだけです。特定の項目を使う場合に必要となる他の項目もあります。それらはこのドキュメントに注釈付きで書かれています。

    マニフェスト内の項目は順序を問いません。ここに書かれていない項目がマニフェストに含まれていても無視されます。

    activities

    (Firefox OS のみ、オプション) アプリが対応している一連の Web Activities を指定します。この項目の各プロパティはアクティビティです。アクティビティ名は自由形式テキストで、各アクティビティはオブジェクトで表されます。以下は share というアクティビティを 1 つ含んだアクティビティ項目の例です。

    "activities": {
  "share": {
    "filters": {
      "type": [ "image/png", "image/gif" ]
    },
    "href": "foo.html",
    "disposition": "window",
    "returnValue": true
  }
}

    この例では、share アクティビティのオブジェクトは、filters、href、disposition、returnValue というプロパティを持っています。これらの意味は アクティビティハンドラの説明 に書かれています。

    appcache_path

    (オプション) アプリケーションキャッシュ (AppCache) マニフェストへの絶対パス。Open Web App がインストールされる際、この AppCache マニフェストが取得、解析され、CACHE ヘッダ以下の静的ファイルがキャッシュされます。

    "appcache_path": "/cache.manifest"

    chrome

    Firefox OS 1.1 ???

    (オプション) 画面下部にナビゲーションインタフェースを追加します。これは独自の戻るボタンを備えていないアプリに多少のナビゲーションコントロールを追加する簡単な方法です。

    ナビゲーションエリアは以下の左側の画像では折り畳まれています。上向きにドラッグすると右側の画像のようになります。

    例:

    "chrome": { "navigation": true }

    注: この項目は最後の手段であるべきです。最高のユーザ体験を提供するため、アプリのインタフェースに戻るボタンを追加することを検討してください。

    csp

    (オプション) アプリの Content Security Policy を指定します。このポリシーはアプリに読み込まれたすべてのページに適用されます。詳しくは Apps の CSP ページを参照してください。

    default_locale

    (locales がある場合は必須) マニフェストの使用言語を示す言語タグ (RFC 4646) です。ここで用いた言語タグは locales 項目に含めないでください。locales を記載しない場合でも default_locale を定義しておくのが良いでしょう。Firefox Marketplace ではこの値からマニフェストの使用言語が認識され、翻訳された項目に適切なロケール値が使用されます。default_locale が定義されていない場合、Marketplace は配信ロケールに基づいて言語を推定しなければなりません。default_locale はアプリをインストールする端末では使用されません。

    英語の例:

    "default_locale": "en"​

    description

    (必須) 人間が読み取れるアプリの説明 (最長 1024 文字)。

    developer

    (オプションですが下記注参照) アプリの開発者に関する情報。以下のプロパティを持ちます。

    • name - 開発者の名前。
    • url - アプリの開発者についての詳しい情報を記載したサイトの URL。この URL は通常、ユーザがダッシュボード (もしくはブラウザ) 内でアプリの詳細情報を表示し、アプリ開発者の名前をクリックしたときに開かれます。

    注: Firefox Marketplace は、アプリの一覧に表示するため、すべての登録アプリに開発者名の記載を求めています。

    fullscreen

    (Firefox OS のみ、オプション) これを true もしくは false に設定することで、ランタイムがそのアプリを全画面モードで起動するかどうかを示します。例:

    "fullscreen": "true"

    icons

    (必須) アイコンの URI に対するアイコンサイズを示すマップ (絶対パス、data URI のいずれか)。アイコンは正方形で、アプリを視覚的に表すことを目的としたものです。/ で始まるパスは、アプリの配信元からの相対パスとして解釈されます。アイコンには、その画像の 4 つの角すべてに広がる無地の背景を敷いてはいけません。

    Firefox OS については、アイコンは アプリアイコンガイドライン に従う必要があり、ドロップシャドウを付けてはいけません。典型的なサイズは、

    128 x 128
    Firefox Marketplace 表示用
    60 x 60
    実機アイコン用。128px のアイコンのみ必須ですが、端末上での表示を最適化するためこのサイズも含めることを推奨します
    32 x 32, 90 x 90, 120 x 120 and 256 x 256
    これらのアイコンは、Windows 7 や Android など、アプリをインストール可能な他の様々なプラットフォームでの表示最適化のために必要です。

    重要: Marketplace 登録時に求められる必須サイズは 128 x 128 のみです。

    複数のアイコンは以下の例のように指定できます。

    "icons": {
  "60": "/img/icon-60.png",
  "128": "/img/icon-128.png"
}

    注: Windows 7 と Android プラットフォームでは次のアプリアイコンサイズに対応しています: 16 x 16、32 x 32、48 x 48、64 x 64、128 x 128、256 x 256

    installs_allowed_from

    (オプション) アプリのインストール実行を許可するひとつ以上のサイト。この項目は、有料アプリ (購入が必要なアプリ) が商取引を行いたいストアを特定する目的で使用することを意図したものです。無料アプリの場合は、この項目を省略することで、どこからでもインストール可能にすることを推奨します。

    これは、<スキーマ名> + <ドメイン> の形式を持つ配信元の配列です。例:

    "installs_allowed_from": [
  "https://marketplace.firefox.com",
  "https://marketplace.example.com"
]

    配列を ["*"] のような形式にした場合、このアプリのインストールはすべてのサイトに許可されます。これが既定値です。なお、空の配列 [] にした場合、自分のサイトを含め、すべてのサイトからのインストールを禁止することになります。

    この項目は、レシートバリデータ が、どのストアがアプリのレシートを発行可能かを管理するために使用できます。

    注: この配列内の URL の末尾にはスラッシュを付けないでください。例えば https://marketplace.example.com/ は正しくありません。末尾にスラッシュを付けるとインストールに失敗します。この問題に関するバグは これ です。

    注: インストール URL のひとつが Firefox Marketplace の場合は、https://marketplace.firefox.com を含めてください。

    launch_path

    (オプション、パッケージ型アプリには必須) アプリの起動時に読み込まれる、アプリの配信元内でのパス。指定されていない場合、アプリの配信元が起動ドメイン URL として扱われます。詳細は パス処理 を参照してください。アプリがドメインのルートディレクトリにない場合は特別な配慮が必要です。

    パッケージ型アプリ では、この項目は、そのパッケージ型アプリを含んだ ZIP ファイルを起点とするコンテンツの開始点を指定します。例えば、launch_path に /myApp/index.html という値が指定された場合、パッケージ型アプリは起動時に /myApp/index.html ファイルを開きます。

    注意: launch_path の値は常にサーバの配信元 (パッケージ型アプリの場合は ZIP ファイルのルート) からの相対パスにする必要があります。配信元やルートに index.html ファイルが置かれている場合でも、この値は単に index.html ではなく /index.html としなければなりません。

    locales

    (オプション) マニフェストに含まれるデータのひとつ以上のロケール固有の上書きを示すマップ。アクセスされた端末のロケールに基づいて UI がローカライズされたビューを提供するのに使用します。例えば、イタリア語圏のユーザがアプリをインストールした場合は、おそらくイタリア語の UI テキストを提供したいと思うでしょう。各ロケールエントリは言語タグ (RFC 4646) をキーとし、マニフェストのスパース表現を含みます。locale 項目内に存在するすべての項目が、マニフェスト内で一致する項目を上書きします。default_locale、locales 自体、installs_allowed_from は上書きできません。これらの項目を上書きするマニフェストは不正なものと見なされます。locales が存在する場合は default_locale も存在しなければなりません。

    この項目のキーは xx-YY あるいは xx といった形式にする必要があります。例えば、pt-BR あるいは es といった形式を使用し、他の pt_br、pt_BR、ES といった形式は避けてください。Firefox Marketplace では一貫性を保つためこのアドバイスが強制されます。

    注: アプリケーションが自分の言語で使用可能かどうかユーザにヒントを与えるため、locales キーを実際のコンテンツのローカライズと一致させることを推奨します。コンテンツのローカライズと locales キーが完全に一致していなくても技術的な問題は生じませんが、ユーザ体験と開発者の評判に関して良くない結果につながる可能性があります。

    スペイン語とイタリア語の例:

    "locales": {
"es": {
  "description": "¡Acción abierta emocionante del desarrollo del Web!",
  "developer": {
    "url": "http://es.mozillalabs.com/"
  }
},
"it": {
  "description": "Azione aperta emozionante di sviluppo di fotoricettore!",
  "developer": {
    "url": "http://it.mozillalabs.com/"
  }
}
}

    messages

    (Firefox OS のみ、オプション) アプリから捕捉可能とするシステムメッセージを示します。これは、システムメッセージとそのメッセージが送信された場合に表示するページのマッピングを記述した配列です。ある Firefox OS アプリからの例を以下に示します。

    "messages": [
  { "telephony-new-call": "/dialer/index.html#keyboard-view" },
  { "bluetooth-dialer-command": "/dialer/index.html#keyboard-view" },
  { "headset-button": "/dialer/index.html#keyboard-view" },
  { "notification": "/dialer/index.html#keyboard-view" },
  { "alarm": "/facebook/fb_sync.html" }
]

    この例では、(電話の着信を示す) telephony-new-call システムメッセージが送信された場合、端末は /dialer/index.html ページの #keyboard-view アンカーを表示します。このページでは、ユーザに着信を通知し、その通話に応答するコントロールを提供するといった処理を行う必要があります。

    注: バグ 800431 が解決するまで、指定されたページはこのメッセージ用に登録されたのと同じでなければなりません。

    name

    (必須) 人間が読み取れるアプリの名称 (最長 128 文字)。

    注: 配布後にアプリの名称を変更した場合、ユーザの端末にインストールされている既存のアプリについては名前が更新されません。

    orientation

    (Android と Firefox OS のみ、オプション) 端末の向きが変わった場合でもアプリを固定する向きを定義する配列。配列内の各エントリは、portrait、landscape、portrait-primary、landscape-primary、portrait-secondary、landscape-secondary のいずれかを取ることができます。-primary と -secondary を含んだオプションは、端末の向きが変わった場合でも、ひとつの端末の向きを固定します。-primary と -secondary を含まないオプションは、プライマリとセカンダリの両方のルールを組み合わせます。追加で -secondary という接尾辞を付けたオプションは、接尾辞なしのオプションから 180 度の回転を暗示します。例えば、スマートフォンを逆さまに持つこと (それでもまだ高さよりも幅が小さいという場合) は portrait-secondary の向きという意味になります。この項目が適切な値の場合、端末の向きが変わった場合でもランタイムはアプリの表示方向を変えません。例:

    "orientation": ["portrait","landscape-secondary"]

    origin

    Firefox OS 1.1 ???

    (特権、公認アプリのみ) 特権あるいは公認のパッケージ型アプリ に 配信元 を与えるために使用されます。ホスト型アプリについては、既に配信元 (アプリが配信されているサイト) が設定されているため、この項目は必要ありません。この項目のひとつの使用例として、OAuth や Persona といった認証機能をパッケージ型アプリへ追加したい場合にそれを容易にすることが挙げられます。

    パッケージ型アプリについて内部的に使用される特別なプロトコルがあります。それは app://<UUID> というもので、<UUID> はアプリがインストールされている端末に固有の長い文字列です。現時点ではアプリ開発者がこの URL を簡単に取得する方法はありません。origin 項目は、app://my-app.com のような、各インストール済みアプリが使用する単独のドメイン名でこの UUID を置き換えられるようにするものです。

    このドメイン名は開発者の管理下にあるドメインでなければなりません。例えば Facebook ではないのに app://facebook.com のようなドメインを使ったアプリはおそらく Firefox Marketplace では承認されないでしょう。

    例 (必ず app:// プロトコルを使ってください):

    "origin": "app://my-app.com"

    permissions

    (オプション) アプリが必要とする一連の許可設定。許可設定は重要な端末 API (WebAPI) へのアクセスを制御するものです。アプリは、ユーザの許可を必要とし、使おうとするすべての API をリストアップしなければなりません。アプリがそうした API を permissions 項目内の対応するエントリなしに使おうとした場合、エラーとなり使用できません。

    注: これらの許可設定の多くは 特権付きアプリと公認アプリ にのみ必要なもので、ホスト型アプリには意味がありません。

    許可設定項目はオブジェクトで、プロパティごとにひとつの許可を指定します。各 API エントリには description プロパティを含める必要があります。一部の API は access プロパティも必要です。permissions 項目の例は以下の通りです。

    "permissions": {
  "contacts": {
    "description": "Required for autocompletion in the share screen",
    "access": "readcreate"
  },
  "alarms": {
    "description": "Required to schedule notifications"
  }
}
    • description - この API の使用を要求する意図を明記した文字列。このプロパティは必須です。
    • access - 許可に必要なアクセスの種類を指定する文字列。このプロパティは一部の API のみ必要です。取り得る値は readonly、readwrite、readcreate、createonly です。

    WebAPI には様々な種類があります。使用可能な許可設定の一覧は アプリの許可設定 を参照してください。

    precompile

    Firefox OS 1.4 ???

    (Firefox OS のみ、オプション、パッケージ型もしくは公認アプリ) アプリ内に含まれている asm.js コードをインストール時にコンパイルし、初回起動時にインストールする必要がある場合に指定します。これにより、最初のインストール時間が長くなる代わりに、初回起動時のアプリ起動コストが大幅に削減されます。この項目には、コンパイルする必要のある asm.js コードを含む JavaScript ファイルへの相対パス配列を記載します。

    "precompile": [
  "game.js",
  "database.js"
]

    redirects

    (Firefox OS のみ、オプション、特権・認定アプリのみ) 特権アプリや認定アプリは外部認証を行う必要があるかもしれません。例えば、Facebook の OAuth 認証を行って、個人の連絡先情報を取得するアプリもあるでしょう。認証が完了すると、認証サーバは通常、アプリ開発者があらかじめ指定した URL へアプリをリダイレクトし返します。

    しかし、特権・認定アプリアプリは、リダイレクト先として指定可能な正規の Web URL を持っていません。なぜならそのアプリはどのサイトにもホストされていないからです (これは公開の配信元を持たない パッケージ型アプリ です)。そのため、redirects 項目を使って、Web URL を内部アプリ URL へリダイレクトします。

    例:

    "redirects": [
  {"from": "http://mydomain.com/oauth2/flow.html",
    "to": "/redirects/redirect.html"},
  {"from": "http://mydomain.com/oauth2/dialogs_end.html",
    "to": "/redirects/dialogs_end.html"}
]

    from には Web URL を、to にはリダイレクト先の内部アプリ URL を設定します。内部アプリ URL は相対 URL でなければならず、URL の前に http:// のようなプロトコルを使うことはできません。

    redirects によって宣言されるリダイレクトの範囲は、それを宣言したアプリ自体に制限されます。これは、複数のアプリが同じ公開 URL を独自のローカルリソースへリダイレクトできるようにするためであり、また特定のアプリケーションによる公開 URL のグローバルハイジャックを防ぐための措置でもあります。

    注: from で定義された URL は、HTTP リダイレクト (30x HTTP ステータスヘッダ) を通じてリクエストされた場合に限って認識され、内部の to パスへ引き渡されます。HTTP リダイレクトなしに URL を開いた場合、redirects 設定は無視されます。

    type

    (オプション) アプリの種類。web、privileged、certified のいずれか。これらのアプリの種類によって、重要な端末 API (WebAPI) へアクセス可能なレベルが異なります。各タイプの説明は以下の通りですが、詳しい説明は パッケージ型アプリ を参照してください。WebAPI へのアクセスについては permissions 項目を参照してください。

    • web - 通常のホスト型アプリ。この種類のアプリには WebAPI への最低限のアクセスが提供されます。マニフェストで type 項目を指定しない場合、web が既定値となります。
    • privileged - Firefox Marketplace などのアプリストアによって承認された、認証済みアプリ。この種類のアプリには web アプリよりも広範な WebAPI へのアクセスが提供されます。
    • certified - スマートフォン上の既定通話アプリやシステム設定アプリなど、重要なシステム機能を備えたアプリ。アプリストアに登録されたサードパーティ製アプリに与えられる権限ではありません。この種類のアプリには WebAPI への最大限のアクセスが提供されます。

    例:

    "type": "privileged"

    version

    (オプション) マニフェストのバージョンを表す文字列。Web ランタイムはこの値を一切使用しませんので、任意の値を取ることができます。 この文字列をマニフェストへ追加し、抽出して、様々な更新のケースに対処するために役立てることもできます。アプリの更新 を参照してください。

    パス処理

    マニフェスト内でパスを持つすべての項目は絶対パス (例えば /images/myicon.png) でなければならず、そのパスはアプリと同じ配信元から提供されるものでなければなりません。

    また、launch_path を設定する場合は、次の 2 通りの方法があります。

    • 例えば mywebapp.github.com/ のように、アプリが Web サーバのルートに保存されている場合は、launch_path を / とします。
    • そうではなく、例えば mymarket.github.com/mywebapp/ のように、アプリがサブディレクトリに保存されている場合は、launch_path を /mywebapp/ とします。

    マニフェストの検証

    マニフェストを検証するには、このサイトを使用してください: https://marketplace.firefox.com/developers/validator

    マニフェストの検証に使用できる API もあります: http://firefox-marketplace-api.readthedocs.org/en/latest/

    マニフェストの配信

    アプリマニフェストはアプリが配信されるのと 同じ配信元 から配信する必要があります。

    マニフェストはファイル拡張子 .webapp 付きで保存しなければなりません。また、Content-Type ヘッダを application/x-web-app-manifest+json として配信する必要があります。これは今のところ Firefox では必須ではありませんが、Firefox Marketplace では必須です。Firefox OS は、ユーザがインストールを実行したページの origin がアプリ自体の origin の異なる場合のみ、この確認を行います。Content-Security-Policy や X-UA-Compatible といった他のヘッダは必要ありません。

    特定の攻撃リスクを軽減するため、SSL 通信を通じてマニフェストを配信することも可能です。マニフェストは HTTP 圧縮 で配信することもできますが、キャッシュされるべきではありません。

    Firefox Marketplace へアプリを登録する場合、マニフェストの文字エンコーディングは UTF-8 とする必要があります。バイトオーダーマーク (BOM) は省略することをお勧めします。他のエンコーディングは、Marketplace では評価されませんが、Content-Type ヘッダの charset パラメータで指定可能です (例: Content-Type: application/x-web-app-manifest+json; charset=ISO-8859-4)。

    ユーザエージェントは、可能な場合、アプリのインストールをユーザに確認する際、サイト識別情報と TLS ステータスを意味のある形で伝えます。

    Apache からの配信

    .htaccess ファイルに以下の行を追加します。

    AddType application/x-web-app-manifest+json .webapp
    注: これは .webapp を拡張子として使っていることを前提としています。.json やその他の拡張子を使う場合は、それに応じてコードを書き換えてください。

    .htaccess ファイルがない場合は、サーバのルートディレクトリに新規作成してください。もし上記のコードを追加しても期待通りに動作しない場合、AddHandler x-web-app-manifest+json .webapp という行も追加する必要があるかもしれません。

    IIS からの配信

    web.config ファイルを編集する必要があります。これはおそらく Web サイトのルートディレクトリにあるでしょう。

    <configuration> <system.webServer> <staticContent> セクションに新しい項目を追加する必要があります。

    <remove fileExtension=".webapp" />
<mimeMap fileExtension=".webapp" mimeType="application/x-web-app-manifest+json; charset=UTF-8" />

    IIS での静的なコンテンツ MIME マッピングの追加に関する詳しい情報

    注: これは .webapp を拡張子として使っていることを前提としています。.json やその他の拡張子を使う場合は、それに応じてコードを書き換えてください。
    nginx からの配信

    設定ディレクトリにある mime.types ファイルを編集する必要があります。これはおそらく /etc/nginx/ か /opt/nginx/ にあります。

    以下のような設定が見つかるはずです。最後の行を追加してください。

    types {
text/html   html htm shtml;
text/css    css;
text/xml    xml;
application/x-web-app-manifest+json   webapp;
}
    注: これは .webapp を拡張子として使っていることを前提としています。.json やその他の拡張子を使う場合は、それに応じてコードを書き換えてください。
    GitHub からの配信

    マニフェストファイルを GitHub Pages から提供する場合、GitHub が自動的に application/x-web-app-manifest+json の Content-Type ヘッダで提供してくれます。マニフェストファイルは .webapp 拡張子を使う必要があります。例: manifest.webapp

    Python からの提供

    Python がインストールされている場合は、Python を起動し、以下のコードを実行することで、ローカルディレクトリから簡単にサーバを実行できます。

    import SimpleHTTPServer
import SocketServer
SimpleHTTPServer.SimpleHTTPRequestHandler.extensions_map['.webapp'] = 'application/x-web-app-manifest+json'
httpd = SocketServer.TCPServer(("", 3000), SimpleHTTPServer.SimpleHTTPRequestHandler)
httpd.serve_forever()
    Rack (Ruby) からの配信

    config/initializers/mime_types.rb に以下の行を追加します。

    Rack::Mime::MIME_TYPES['.webapp'] = 'application/x-web-app-manifest+json'
